# 常见问题 - ID生成规则体系

> 面试高频问题：你们业务中付款单据的单据编号、业务ID等的生成规则是什么样的？为什么？

## 一、我们的ID生成规则体系

### 1. 付款单据编号（对用户可见）

**规则格式：**
```
PAY + YYMMDD + 4位企业编码 + 6位流水号
示例：PAY2403150001001234
```

**各段含义：**
- `PAY` - 业务类型标识（付款Payment）
- `240315` - 日期（2024年3月15日）
- `0001` - 企业编码（集团内子公司识别码）
- `001234` - 当日流水号

**生成代码示例：**
```java
@Component
public class PaymentNoGenerator {

    @Autowired
    private StringRedisTemplate redisTemplate;

    private static final String REDIS_KEY_PREFIX = "PAYMENT_NO:";

    /**
     * 生成付款单号
     */
    public String generatePaymentNo(Long enterpriseId) {
        // 1. 获取当前日期
        String dateStr = LocalDate.now().format(DateTimeFormatter.ofPattern("yyMMdd"));

        // 2. 格式化企业ID（不足4位补零）
        String enterpriseCode = String.format("%04d", enterpriseId % 10000);

        // 3. 从Redis获取当日序列号（原子递增）
        String redisKey = REDIS_KEY_PREFIX + dateStr + ":" + enterpriseCode;
        Long sequence = redisTemplate.opsForValue().increment(redisKey);

        // 4. 设置过期时间（2天）
        redisTemplate.expire(redisKey, 48, TimeUnit.HOURS);

        // 5. 格式化流水号（不足6位补零）
        String sequenceStr = String.format("%06d", sequence);

        // 6. 拼接完整单号
        return "PAY" + dateStr + enterpriseCode + sequenceStr;
    }
}
```

### 2. 业务ID（内部使用，数据库主键）

**规则：** 使用**雪花算法**（Snowflake）

```java
@Component
public class BusinessIdGenerator {

    private final SnowflakeIdWorker idWorker;

    public BusinessIdGenerator() {
        // 使用Redis分配workerId，支持多实例部署
        long workerId = getWorkerIdFromRedis();
        this.idWorker = new SnowflakeIdWorker(workerId);
    }

    /**
     * 生成业务ID
     */
    public Long generateBizId() {
        return idWorker.nextId();
    }
}
```

### 3. 批量付款批次号

**规则：**
```
BATCH + YYMMDD + 6位批次号
示例：BATCH240315000001
```

**特点：** 支持批量操作，便于对账

## 二、为什么这样设计？（4个核心原因）

### 原因1：业务可读性（财务审计需求）

```java
// 财务人员在Excel中筛选和分析时：
PAY2403150001001234 → 一眼看出是2024-03-15的付款单，企业0001
PAY2403150002005678 → 同一天，企业0002的单据

// 对比纯数字ID：
7891234567890 → 完全看不出业务含义
```

**业务价值：**
- 对账时快速定位（银行回单对账）
- 快速识别问题单据所属企业
- 财务报表按日期归类

### 原因2：高性能（支撑高并发）

**Redis原子递增的优势：**
```java
// 性能对比
数据库自增ID:   300-500 TPS（需要加锁、磁盘IO）
Redis原子递增:  10000+ TPS（内存操作+单线程无竞争）

// 代码中的高性能实现
Long sequence = redisTemplate.opsForValue().increment(redisKey);
// Redis单线程模型保证原子性，无需分布式锁
```

**冬季（高峰期）双11场景：**
- 企业0001在15:00:00同时提交1000笔付款
- Redis单节点轻松支撑，无需分布式锁
- 生成1000个单号耗时 < 10ms

### 原因3：唯一性和防重性

**防止重复提交的完整方案：**
```java
@Service
public class PaymentService {

    @Autowired
    private PaymentNoGenerator paymentNoGenerator;

    @Autowired
    private RedissonClient redissonClient;

    @Transactional(rollbackFor = Exception.class)
    public PaymentResult createPayment(PaymentReq req) {
        // 1. 生成单号
        String paymentNo = paymentNoGenerator.generatePaymentNo(req.getEnterpriseId());

        // 2. Redis分布式锁，防止前端重复提交
        String lockKey = "PAYMENT_CREATE:" + req.getEnterpriseId() + ":" + req.getBizKey();
        RLock lock = redissonClient.getLock(lockKey);

        try {
            if (lock.tryLock(5, TimeUnit.SECONDS)) {
                // 3. 幂等性检查（防止重复）
                Payment existing = paymentMapper.selectByBizKey(req.getBizKey());
                if (existing != null) {
                    return PaymentResult.of(existing);
                }

                // 4. 创建付款单
                Payment payment = buildPayment(req, paymentNo);
                paymentMapper.insert(payment);

                return PaymentResult.success(paymentNo);
            } else {
                throw new BizException("请勿重复提交");
            }
        } finally {
            lock.unlock();
        }
    }
}
```

**唯一性保障机制：**
- Redis原子递增 → 保证序号唯一
- 日期+企业编码 → 分片隔离，避免跨企业冲突
- 分布式锁 → 防止同一业务重复生成
- 数据库唯一索引 → 最后一道防线

### 原因4：便于分库分表

**分片策略：**
```java
// 单据表按企业编码分库分表
payment_0001_202403  // 企业0001在2024-03的数据
payment_0002_202403  // 企业0002在2024-03的数据

// 分片键就是单据号中的企业编码
// SQL查询可以直接定位到对应分片
SELECT * FROM payment_0001_202403
WHERE payment_no = 'PAY2403150001001234';
```

**优势：**
- 避免分片键计算（直接从单号提取）
- 支持范围查询（按日期、企业）
- 便于数据归档（按月份归档历史数据）

## 三、ID生成方案的演进和踩坑

### 初期方案（V1.0）：数据库自增ID

```java
// 问题代码（不要这样做）
@Transactional
public String generatePaymentNo() {
    // 查询当前最大值
    String maxNo = paymentMapper.selectMaxPaymentNo();
    // +1生成新单号
    return increment(maxNo);
}
```

**遇到的问题：**
1.  **「并发重复单号」**  - 事务隔离导致两个请求读到相同最大值
2.  **「性能瓶颈」**  - 每次生成都要查询数据库，500 TPS撑不住
3.  **「分库分表后重复」** - 多个库生成相同单号

### 优化方案（V2.0）：号段模式

```java
@Component
public class SegmentIdGenerator {

    private Long currentId = 0L;
    private Long maxId = 0L;

    @Synchronized  // 单机锁
    public Long nextId() {
        if (currentId >= maxId) {
            // 批量从数据库获取段
            fetchNextSegment();
        }
        return currentId++;
    }
}
```

**优缺点：**
- ✅ 减少数据库访问
- ❌ 还是有单点瓶颈
- ❌ 应用重启会丢失一段号码

## 四、生成规则的11个要点

| 要点 | 决策 | 原因 |
|------|------|------|
| **长度** | 18位 | 兼顾可读性和存储 |
| **前缀** | PAY | Payment业务标识 |
| **日期格式** | YYMMDD | 6位节省空间，足够百年 |
| **企业编码** | 4位 | 支持10000家企业 |
| **流水号** | 6位 | 每企业每天100万单 |
| **分隔符** | 无 | 节省空间，适用于URL参数 |
| **校验位** | 无 | 内部系统，信任源数据 |
| **生成原子性** | Redis INCR | 比数据库和分布式锁快 |
| **过期时间** | 48小时 | 兼顾跨天处理和内存占用 |
| **workerId分配** | Redis | 支持动态扩缩容 |
| **回拨时间** | 不能回拨 | 雪花算法时间回拨导致ID重复 |

## 五、针对不同场景的对比

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| **C端（用户可见）** | 可读编码 | 用户投诉时能快速定位 |
| **B端（企业内部）** | 可读编码 | 财务审计、对账需求 |
| **内部系统ID** | 雪花算法 | 高并发、高性能 |
| **日志/追踪ID** | 雪花算法 | 无需可读性，追求性能 |
| **测试环境** | UUID | 简单，无需考虑性能 |
| **一次性链接** | 短UUID | 防猜测，简短 |

## 六、面试回答（3分钟版本）

### 问题：你们业务中付款单据的单据编号、业务ID等的生成规则是什么样的？为什么？

**回答：**

我们财资司库平台采用了分层ID生成策略：

1. **付款单据编号**（对用户可见）：
   ```
   PAY + YYMMDD + 4位企业编码 + 6位流水号
   ```
   **原因：**
   - **业务可读性**：财务审计、对账时能快速识别
   - **防止重复**：Redis原子递增保证唯一性
   - **分库分表**：企业编码作为分片键

2. **内部业务ID**（数据库主键）：
   **雪花算法**生成，原因：
   - 高性能：支持10000+ TPS
   - 分布式：多实例无冲突
   - 趋势递增：索引效率高

3. **技术实现：**
   ```java
   // 关键代码：Redis原子递增
   Long sequence = redisTemplate.opsForValue().increment(redisKey);
   ```

4. **演进过程：**
   数据库自增 → 号段模式 → Redis原子递增
   原因：从单点瓶颈 → 性能提升 → 分布式支持

5. **防重机制：**
   Redis分布式锁 + 业务唯一键 + 数据库唯一索引三道防线

**这样设计既满足了财务审计的可读性要求，又支撑了高并发场景的性能需求，还考虑了未来分库分表的扩展性。**
